  <action>
Create prisma/seed.minimal.ts — a standalone TypeScript script using Prisma client directly. Run it once to seed development data.

The script must:
1. Import { PrismaClient } from '@prisma/client'
2. Instantiate prisma client (separate from backend singleton — this is a CLI script)
3. Use prisma.wilayah.upsert or createMany with skipDuplicates to be idempotent
4. Seed the following 5 wilayah records (minimum for dropdown testing in Phase 1):
   - Provinsi: "DI Yogyakarta", Kabupaten: "Kota Yogyakarta", Kecamatan: "Mergangsan", Kelurahan: "Wirogunan", kodePos: "55151"
   - Provinsi: "DI Yogyakarta", Kabupaten: "Kota Yogyakarta", Kecamatan: "Mergangsan", Kelurahan: "Brontokusuman", kodePos: "55153"
   - Provinsi: "DI Yogyakarta", Kabupaten: "Kota Yogyakarta", Kecamatan: "Umbulharjo", Kelurahan: "Muja Muju", kodePos: "55165"
   - Provinsi: "DI Yogyakarta", Kabupaten: "Sleman", Kecamatan: "Depok", Kelurahan: "Caturtunggal", kodePos: "55281"
   - Provinsi: "Jawa Tengah", Kabupaten: "Kota Semarang", Kecamatan: "Semarang Tengah", Kelurahan: "Sekayu", kodePos: "50134"

5. For each upsert, use a deterministic where condition. Since Wilayah has no @unique constraint on the combination, use createMany with skipDuplicates: true instead.
6. Log count after seeding: console.log(`Seeded X wilayah records`)
7. Call $disconnect() in finally block

Script structure:
  import { PrismaClient } from '@prisma/client'

  const prisma = new PrismaClient()

  const wilayahData = [ ... ] // 5 records above

  async function main() {
    const result = await prisma.wilayah.createMany({
      data: wilayahData.map(w => ({
        id: crypto.randomUUID(),
        ...w,
      })),
      skipDuplicates: false,  // Always insert; script is meant to run once
    })
    console.log(`Seeded ${result.count} wilayah records`)
  }

  main()
    .catch((e) => { console.error(e); process.exit(1) })
    .finally(async () => { await prisma.$disconnect() })

Note: crypto.randomUUID() is available in Node.js 18+.

After creating the file, run the seed inside the backend container (which has ts-node and DATABASE_URL):
  docker compose exec -T sispos-backend npx ts-node --project tsconfig.json /app/prisma/seed.minimal.ts

Wait, the prisma/ directory is mounted at /app/prisma/ in the backend container (from docker-compose.yml volume mount). The seed file will be accessible at /app/prisma/seed.minimal.ts.

Run the seed command:
  docker compose exec -T sispos-backend npx ts-node --project ./tsconfig.json ./prisma/seed.minimal.ts

Expected output: "Seeded 5 wilayah records"

If ts-node errors with "Cannot find module @prisma/client": the Prisma client was generated during Docker build (RUN npx prisma generate in Dockerfile). It should be in node_modules. If not, run:
  docker compose exec -T sispos-backend npx prisma generate

Then retry the seed command.

Verify seed worked:
  docker compose exec -T sispos-db psql -U sispos_user -d sispos -c "SELECT COUNT(*) FROM wilayah;"

Expected: 5 (or more if run multiple times, since skipDuplicates is false — acceptable for dev seed)

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| CLI → Docker daemon | Migration commands run via docker compose exec; DATABASE_URL never echoed |
| Backend container → PostgreSQL | prisma migrate dev uses DATABASE_URL env var set in container |
| Seed script → PostgreSQL | ts-node seed.minimal.ts uses same PrismaClient |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00-04-A | Information Disclosure | DATABASE_URL in docker exec output | mitigate | docker compose exec -T does not echo env vars in terminal. DATABASE_URL set as container env var, not passed as CLI argument. |
| T-00-04-B | Injection | prisma migrate dev generates migration SQL | accept | Prisma generates migration from schema diff — no hand-crafted SQL injection risk in Phase 0. |
| T-00-04-C | Elevation of Privilege | prisma migrate dev runs as DB owner | accept | Development environment only. In production (Phase 7+ deploy), migration would run as limited-privilege user. |
| T-00-04-D | Tampering | Seed data overwriting existing data | accept | seed.minimal.ts is dev-only; runs once; no production data exists yet. Full seed in Phase 7 uses separate seed.ts and seed.demo.ts. |
| T-00-04-E | Information Disclosure | .env file with real credentials | mitigate | .gitignore excludes .env; devs must create .env locally from .env.example. |
| T-00-SC | Tampering | npm/pip/cargo installs (no new installs in this plan) | accept | No new package installs in Plan 00-04; all deps installed in Plans 00-02 and 00-03. |
</threat_model>
