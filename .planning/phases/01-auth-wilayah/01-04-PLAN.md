<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| User browser → RegisterPage form | Untrusted NIK, phone, password; frontend Zod is UX-only; server enforces authoritative validation |
| sessionStorage ('reg_ponsel') → VerifikasiOtpPage | Phone number stored in sessionStorage between register and OTP screens |
| VerifikasiOtpPage → POST /api/auth/otp/verify | OTP digits assembled client-side; server validates against DB record with TTL |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-01 | Information Disclosure | sessionStorage key 'reg_ponsel' stores nomorPonsel | accept | sessionStorage is tab-scoped, cleared on tab close; only phone number stored (not NIK or password); backend returns masked phone for OTP page display only |
| T-04-02 | Spoofing | Client-side registerSchema bypass via curl/Postman | mitigate | Frontend Zod is UX enhancement only; authoritative enforcement in backend auth.service.ts (per T-01-01 in 01-01); no security computation on client |
| T-04-03 | Tampering | OTP digit state manipulation via browser devtools | accept | OTP validated server-side exclusively; frontend sends raw 6-char string; no crypto or security logic runs in browser |
| T-04-04 | Information Disclosure | NIK visible in DOM input field | accept | NIK is a public identifier per UU KTP; password uses type="password" with show/hide toggle controlled by aria-pressed; no sensitive data in URL query params |
| T-04-05 | Denial of Service | Rapid resend OTP clicks triggering BullMQ floods | mitigate | Frontend enforces max 3 resends with counter in component state; countdown gate prevents back-to-back requests; backend invalidates old OTPs on each resend (per auth.service.ts sendOtp) |
| T-04-SC | Tampering | npm install react-hook-form zod @hookform/resolvers + shadcn components | mitigate | All packages pre-approved in 01-01 Task 0 (blocking human checkpoint); 01-04 depends on 01-01, ensuring Task 0 ran before this plan executes |
</threat_model>
